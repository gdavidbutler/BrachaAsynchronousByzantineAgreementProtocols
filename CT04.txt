Model and tools
---------------

n servers {P_1, ..., P_n} and an open set of clients {C_1, C_2, ...}.
The adversary is t-limited (controls at most t servers, any number of
clients), static, and benign w.r.t. delivery (every message between
honest parties is eventually delivered).  Every server pair has a
secure asynchronous channel (privacy + authenticity, adversary
schedules); every server-client pair likewise.

Protocol instances are tagged with a globally unique ID.  Local
events are input actions (ID, in, type, ...) and output actions
(ID, out, type, ...); other messages are (ID, type, ...).  Once an
honest party performs an output action for a protocol instance, that
instance terminates for that party (all threads stop) - this is the
"have terminated" guard our ct04Dsp.c wrapper enforces as a C-side
early-exit before the dispatch.


AVID (Definition 1, paraphrased)
--------------------------------

A (k, n)-asynchronous verifiable information dispersal scheme is a
pair of protocols (Disperse, Retrieve) such that for any t-limited
adversary, any ID, and any client C_i starting Disperse for ID:

  Termination.   If C_i is honest, all honest servers eventually
                 complete the dispersal (output "stored").
  Agreement.     If some honest server completes the dispersal, then
                 all honest servers eventually do.
  Availability.  If k honest servers complete the dispersal and an
                 honest client C_j starts Retrieve for ID, C_j
                 eventually reconstructs some F'.
  Correctness.   If k honest servers complete the dispersal, there
                 exists a fixed G such that (1) if C_i is honest and
                 dispersed F, G = F; (2) if any honest C_j
                 reconstructs F', G = F'.

Storage blow-up.  Lower bound n/(n - t): the scheme must allow any
n-t honest servers to reconstruct the file, so each server stores at
least |F|/(n-t).


3.3 The main scheme (AVID)
--------------------------

Encoding.  The client encodes F as [F_1, ..., F_n] via a (k, n)-
erasure code over GF(q), with t + 1 <= k <= n - 2t.  It computes the
fingerprint vector D := [H(F_1), ..., H(F_n)] using a collision-
resistant hash H, and sends each P_j the pair (D, F_j).  Two rounds
of all-to-all exchanges follow; messages from P_i to P_j carry (D,
F_i).  A server accepts only blocks consistent with D.  When the
servers agree on D, each P_j stores D in Verify[ID] and F_j in
Data[ID] and emits (ID, out, stored).

Disperse, server P_i.  Per-tag state: e_D := 0, r_D := 0, A_D := {}.

  upon (ID, in, disperse, F) at client C_i:
    encode F into [F_1, ..., F_n]; D := [H(F_1), ..., H(F_n)]
    for j in [1, n]: send (ID, send, D, F_j) to P_j

  upon (ID, send, D, F_i) for the first time:
    if H(F_i) = D_i:
      for j in [1, n]: send (ID, echo, D, F_i) to P_j

  upon (ID, echo, D, F_m) from P_m for the first time:
    if H(F_m) = D_m:
      A_D := A_D union {(m, F_m)}; e_D := e_D + 1
      if e_D = max(ceil((n+t+1)/2), k) and r_D < k:
        interpolate polynomial f-bar of degree <= k - 1 from A_D
        F-bar_j := f-bar(j) for all j in [1, n]
        if H(F-bar_j) = D_j for all j:
          for j in [1, n]: send (ID, ready, D, F-bar_i) to P_j
        else:
          output (ID, out, abort)

  upon (ID, ready, D, F_m) from P_m for the first time:
    if H(F_m) = D_m:
      A_D := A_D union {(m, F_m)}; r_D := r_D + 1
      if e_D < max(ceil((n+t+1)/2), k) and r_D = k:
        interpolate f-bar as above; if all H(F-bar_j) = D_j:
          for j in [1, n]: send (ID, ready, D, F-bar_i) to P_j
        else:
          output (ID, out, abort)
      else if r_D = k + t:
        Verify[ID] := D; Data[ID] := F-bar_i
        output (ID, out, stored)

The two interpolation branches (echo-side and ready-side) are
mutually exclusive by counter intervals: r_D < k vs r_D = k.  At
t >= 1, "r_D = k" and "r_D = k + t" are also disjoint by counter
value, so the else-if structure of the READY handler matters only
at the degenerate t = 0 case (outside Theorem 3's t + 1 <= k regime).

Retrieve, client + server.

  upon (ID, in, retrieve, ID') at client C:
    for all j: send (ID, retrieve, ID') to P_j
    wait for k messages (ID, block, F'_j, D) from distinct servers
       with the same D and D_j = H(F'_j)
    interpolate f' of degree <= k - 1 from { (j, F'_j) : j in J }
    output (ID, out, retrieved, F' = [f'(1), ..., f'(k)])

  upon (ID, retrieve, ID') from C_m at server P_i:
    if Data[ID'] != bot:
      send (ID, block, Data[ID'], Verify[ID']) to C_m

Theorem 3.  AVID is a (k, n)-asynchronous verifiable information
dispersal scheme for t + 1 <= k <= n - 2t.  For k = n - 2t the
storage blow-up of Disperse is 3 + o(1) and the communication blow-
up is O(n) for Disperse, O(1) for Retrieve.


3.4 Analysis
------------

Lemma 4 (standard, [Bra84]).  If an honest P_i sends a ready-message
with D^(i) and an honest P_j sends a ready-message with D^(j), then
D^(i) = D^(j).  (Two honest peers can only ever ready-broadcast the
same root commitment, hence the protocol's reliable broadcast on D.)

Termination, Agreement, Availability, Correctness sketches.

  Termination.  Trivial under an honest dealer: the only way a
  malicious server can prevent acceptance is producing a hash
  collision (negligible).

  Agreement.  Define (ID, ready, D, F_m) valid iff D_m = H(F_m).  A
  server that completes the dispersal with Verify[ID] = D has
  received k + t valid ready-messages with that D, so at least k of
  them came from honest servers.  Those k honest servers each
  reach-broadcast a valid ready with D, so every honest server sees
  k valid ready-messages with the same D; by the inductive argument
  in the paper, every honest server then sends a valid ready-message
  and completes (or, with negligible probability, the adversary
  found a collision).

  Availability.  k honest servers store the same D, so the client's
  Retrieve gets k consistent blocks and reconstructs.

  Correctness.  The paper's Lemma 5 / hash-collision argument
  ensures that whatever k blocks the client interpolates from, they
  reconstruct the dealer's F (under an honest dealer) or all honest
  servers reconstruct the same G (under a Byzantine dealer).


3.5 AVID-H (Merkle-tree variant) - the version we plug in
---------------------------------------------------------

§3.5 reduces communication and storage by replacing the n-entry
fingerprint vector D with a single Merkle root h_r of the same
[H(F_1), ..., H(F_n)] hashes.  Per-block authenticity is checked via
a per-block fingerprint FP_i (the sibling-hash path from leaf i to
root) and the predicate verify(i, F_i, FP_i, h_r):

  verify(i, F_i, FP_i, h_r):
    h := H(F_i)
    for j := log_2(n) downto 1:
      h := H(h, FP_i[j]) or H(FP_i[j], h)
        depending on the unique path from leaf i to the root
    return h = h_r

Lemma 5.  Let H be collision-resistant.  If F_i and F'_i with
fingerprints FP, FP' both pass verify(i, ., ., h_r) for the same
h_r, then F_i = F'_i (except with negligible probability).

The improved scheme is obtained by modifying AVID as follows:

  - The client, after encoding, computes the hash tree of [F_1, ...,
    F_n] and uses the root hash h_r in place of D.
  - send / echo / ready messages carry (h_r, FP_i, F_i) instead of
    (D, F_i).
  - Counters e_*, r_*, the set A_* are indexed by h_r, not by D.
  - The "H(F_i) = D_i" check is replaced by verify(i, F_i, FP_i,
    h_r).  When a server reconstructs F-bar_j, it also reconstructs
    the corresponding path-hashes and uses verify to check.
  - Storage is FP(i) + h_r per server, plus the data block.

Theorem 3 carries through under Lemma 5.  Communication complexity
improves to O(n^2 |F|/k + n^2 log n |H|); storage to n|F|/k +
O(n log n |H|).  For k = n - 2t, communication blow-up is O(n).

How we plug this in.  Our ct04Dsp.dtc lives in the plugin-abstract
restatement of §3.5: piece-valid is the plugin's per-block
predicate (whatever shape it takes - hash-vector D, Merkle root h_r,
or future schemes), decode-and-verify-all-pieces is the plugin's
re-interpolate-and-re-check predicate, and the wire-message
vocabulary (send / echo / ready) is paper-direct.  The thrDsp.h
contract names exactly these capabilities; thrDspRsec (RS) and
thrDspSss (Shamir) ship as adapters.


3.6 A communication-efficient protocol for reliable broadcast
-------------------------------------------------------------

Theorem 6.  Provided t < n/3, there exists an asynchronous reliable
broadcast protocol with message complexity O(n^2) and communication
complexity O(n |m| + n^2 log n |H|).

Construction: run AVID-H's Disperse on F := m, but at the point
each server would store its block F-bar_i in Data[ID], instead
output the whole reconstructed file F-bar = [F-bar_1, ..., F-bar_k]
as the delivered message m'.  No new wire rules - same Disperse
dispatch, different wrapper output.  Servers do not need to run
Retrieve at all in this mode.

This is the "RBC-from-AVID-H" deployment shape: it sits on the
same ct04Dsp dispatch and is a choice of what the C wrapper hands
back on the (ID, out, stored) event.


3.7 AVID-RBC outer wrap (the optimal-storage-blow-up scheme)
------------------------------------------------------------

The §3.2 simple scheme already achieves asymptotically optimal
storage blow-up n/(n-t) + o(1) by combining an (n-t, n)-erasure
code with a reliable broadcast for the n block hashes.  §3.2 used
Bracha 1984's reliable broadcast.  §3.7 replaces it with the §3.6
RBC-from-AVID-H result, yielding a scheme with O(n |F|) total
communication AND optimal storage blow-up.

Two distinct k values are in play:

  k_inner = n - 2t   for the inner Disperse (the §3.6 RBC)
  k_outer = n - t    for the outer (n-t, n) erasure code that gives
                     the optimal storage blow-up

After the inner RBC delivers the whole file F-bar to every honest
server, each server:

  1. Encodes F-bar with the (k_outer, n)-erasure code into n outer
     blocks G_1, ..., G_n.
  2. Computes the flat hash list [H(G_1), ..., H(G_n)].
  3. Keeps G_self and the hash list; erases the other G_j.
  4. Outputs (ID, out, stored).

A cost: erasure-coding is applied twice (once inside the inner
Disperse for distribution; once outside for storage).  This is the
"computational cost" the paper notes (p. 13).

The hash list is FLAT (not Merkle-tree-rooted) at the outer layer
because every server has the same list and there is no longer a
need for compact per-piece authenticators - the outer pieces are
not redistributed.  This is why our ct04DspRbc.[hc] takes a
plain hash callback and a flat erasure-encoder callback instead of
re-using the thrDsp plugin (which targets the Merkle-tree case).
